# Requirements Capture

## 1. Problem Frame

当前需求是基于 `iOfficeAI/AionUi` 做二次开发，在不改动官方既有功能的前提下，新增 `公众号` 与 `外贸` 两个业务域入口，并统一为“对话驱动业务动作”模式。该需求需要在后续持续同步上游更新时保持低冲突、可回归、可演进。

本次文档目标是把 U1 从“方向描述”细化为可验证条目，并按优先级分为“必须/可选/暂缓”，为后续 `ae-plan` 与 `ae-work` 提供稳定输入。

## 2. Goals

- 把“公众号与外贸统一对话驱动”拆解为明确需求清单，减少口径漂移。
- 明确哪些条目是本轮必须完成，哪些可以延后。
- 为最小 PoC 闭环建立验收标准（公众号：生成文章->推送草稿箱；外贸：查询商品->上架商品）。
- 明确高风险动作的安全底线（确认、鉴权、幂等、审计）。

## 3. Non-Goals

- 不在本阶段落地具体公众号平台 SDK 实现细节。
- 不在本阶段落地具体外贸平台 SDK/ERP 对接细节。
- 不在本阶段引入完整生产级 CI/CD 发布自动化。
- 不在本阶段重构 AionUi 核心会话引擎。

## 4. Requirements

说明：`必须` = 当前阶段必须达成；`可选` = 不影响首轮闭环但有价值；`暂缓` = 明确延后到后续阶段。

| ID | 分类 | 需求条目 | 约束/说明 |
|---|---|---|---|
| RQ-M01 | 必须 | 新增侧边栏 `公众号`、`外贸` 两个入口 | 不影响原生菜单与原功能路径。 |
| RQ-M02 | 必须 | 两个入口均进入域化对话上下文 | 必须能区分 `domain=wechat` 与 `domain=trade`。 |
| RQ-M03 | 必须 | 公众号与外贸统一对话动作协议 | 最低包含 `domain/action/payload/requiresConfirmation/idempotencyKey`。 |
| RQ-M04 | 必须 | 所有高风险写操作必须二次确认 | 未确认不得执行发布/上架/删除类动作。 |
| RQ-M05 | 必须 | 写操作必须具备幂等键 | 同一 `idempotencyKey` 重试不得重复写入。 |
| RQ-M06 | 必须 | 域级权限隔离 | `wechat` 凭据不能执行 `trade` 写动作，反之亦然。 |
| RQ-M07 | 必须 | 动作全链路审计 | 每次动作可关联 `traceId`，可追溯“会话->确认->执行->结果”。 |
| RQ-M08 | 必须 | 保持“薄壳扩展”改动边界 | 新增逻辑优先放扩展目录，尽量不侵入核心引擎。 |
| RQ-M09 | 必须 | 支持上游持续同步策略 | 至少具备可执行的同步节奏与冲突审计机制。 |
| RQ-M10 | 必须 | 公众号 PoC 闭环可演示 | 对话生成文章并推送到草稿箱。 |
| RQ-M11 | 必须 | 外贸 PoC 闭环可演示 | 对话查询商品并完成上架动作。 |
| RQ-O01 | 可选 | 域入口权限可见性控制 | 基于角色控制菜单是否显示。 |
| RQ-O02 | 可选 | 域内快捷模板提示 | 提供常用 Prompt 模板提升首轮可用性。 |
| RQ-O03 | 可选 | 批量动作确认 UX 优化 | 支持多条写操作合并确认。 |
| RQ-O04 | 可选 | 上游同步辅助脚本 | 自动生成“落后提交数+冲突清单”。 |
| RQ-D01 | 暂缓 | 自动配图、定时发布等公众号增强能力 | 不影响首轮验证目标。 |
| RQ-D02 | 暂缓 | 外贸批量上下架与库存联动 | 不影响首轮验证目标。 |
| RQ-D03 | 暂缓 | 生产级 CI/CD 发布门禁自动化 | 先以手工清单+冒烟验证过渡。 |
| RQ-D04 | 暂缓 | KMS/HSM 托管密钥体系 | 当前先定义接口与边界，不强行落地。 |

## 5. Acceptance Criteria

### A. 必须项验收（阻断）

- AC-M01: 点击 `公众号` 与 `外贸` 菜单可分别进入对应域会话，且原有菜单行为无回归。
- AC-M02: 两个域的动作请求都走统一协议，字段完整且结构一致。
- AC-M03: 未经二次确认的高风险写动作必定被拒绝执行。
- AC-M04: 同一写操作重复提交时，幂等机制能阻止重复执行。
- AC-M05: 跨域凭据调用被拒绝，并返回可识别的权限错误。
- AC-M06: 任意写动作都可追溯到 `traceId` 与确认证据。
- AC-M07: 公众号与外贸各至少 1 条写入闭环可稳定演示成功。
- AC-M08: 能说明当前版本与上游差异与同步策略，且有可执行回归清单。

### B. 可选项验收（非阻断）

- AC-O01: 可按角色控制域入口展示与隐藏。
- AC-O02: 域内提供至少 2 条高频任务模板并可直接触发对话。
- AC-O03: 批量确认流程减少重复确认交互。
- AC-O04: 同步辅助脚本可输出基础状态报告。

### C. 暂缓项验收（后续里程碑）

- AC-D01: 对应条目进入后续计划时，再补充独立验收标准与风险评估。

## 6. Open Questions

- OQ-01: 两个域入口是否需要按租户隔离展示，还是仅按用户角色控制。
- OQ-02: 写操作确认的有效期与撤销窗口如何定义（例如 60 秒内有效）。
- OQ-03: `machineCode` 错误码是统一枚举还是域内扩展子码。
- OQ-04: 上游同步频率是固定“每周一次”还是按发布节奏动态调整。

## 7. Planning Notes

- 本文可直接作为 `ae-plan` 第二轮输入，优先推进下列最小链路：
  - U2（上游同步策略）+ U4（域入口骨架）+ U5（统一协议）。
- `必须` 项用于定义本轮交付边界，`可选/暂缓` 项默认不阻断首轮上线判定。
- 后续如出现需求冲突，优先以本文件中 `必须` 条目和验收标准作为仲裁基线。
